## Goal

Turn `index.html`'s bare front door (`#screen-signin` in email-mode) from a bare "Sign in to your business" card into a real marketing landing page, while leaving the sign-in mechanism completely untouched.

## Why it's safe to market not-yet-built features (owner's explicit call)

The owner was told plainly, per feature, before this was folded in: self-booking and turn-algorithm/skill-matching don't exist in the code today (verified against `calendar.js` and `turns.js`); AI report insights doesn't exist yet either (the only related code is a manual "copy → paste into your own ChatGPT/Claude tab" button in `reports.js`, and the owner says an Anthropic key exists to eventually replace it). The owner chose, explicitly and after that correction, to have the page describe the intended launch-time feature set rather than only today's shipped state. This plan honors that by keeping every "not yet real" claim inside a click-through detail panel, clearly marked (an amber-toned "Launching soon: ..." line), never on the initial teaser — so the page reads confidently at a glance but stays honest for anyone who reads further, including whoever verifies the punch list before the page actually goes live.

## Request a Live Demo — new, real, small backend feature

The owner wants a distinct path for prospects who want a human-guided look before committing: contact info + "what are you looking for" → the owner personally follows up by video call. This is **not** a relabeled `signup.html` — investigated the real signup pipeline first so this mirrors an already-hardened pattern instead of inventing a new one:

- `signup.html`/`signup.js` POSTs to `/signup/request`, which the Worker forwards clean (IP-only, no salon header) to the reserved **registry Durable Object**, where `cloudflare/signup-util.js`'s `validateSignupRequest()` validates it, per-IP/per-email rate limits apply (5/hr/IP, 200 pending total, 3 pending/email), and it's stored as `signup:<uuid>` with `status:'pending'` for the owner to review and provision manually via `operator.html`'s existing "Pending requests" list — no automatic email either direction; review and follow-up are both manual today.
- **The demo-request flow copies this shape exactly**, simplified (no password/account/slug provisioning — it's a lead, not an account):
  - **NEW `cloudflare/demo-util.js`** — `validateDemoRequest(body)`, mirroring `validateSignupRequest()`'s exact style: `name` (1–60 chars), `email` (same regex/length check), `phone` (optional, ≤40 chars), `lookingFor` (1–500 chars, required — it's the whole point of the form).
  - **NEW `test/demo-util.test.js`** — mirrors `test/signup-util.test.js` exactly (accepts a good request, rejects each bad-input case).
  - **`cloudflare/worker.js`** — import `validateDemoRequest`; add `/demo/request` to the two existing public-route exemption checks (the same two lines that already exempt `/signup/request`, nothing broader); add a top-level forwarder to `registryStub(env)` mirroring the `/signup/request` forwarder exactly; add a DO-side handler mirroring the `/signup/request` handler (own rate-limit key prefix `dreqrl:` so demo-request abuse can't throttle real signups or vice versa), storing `demo:<uuid>` → `{id, status:'pending', name, email, phone, lookingFor, createdAt, ip}` — no password hashing, no slug allocation, no `ownerRecord`.
  - **NEW `demo.html`** — mirrors `signup.html`'s structure exactly (same head boilerplate/fonts/Tailwind config): Name, Email, Phone (optional), "What are you looking for?" (textarea), submit → success card ("Thanks — we'll be in touch to schedule your live demo").
  - **NEW `js/app/demo.js`** — mirrors `signup.js`'s `submitSignup()` exactly: `submitDemoRequest()` POSTing `{name, email, phone, lookingFor}` to `/demo/request`.
  - **`operator.html`** — extend the existing "Pending requests" view with a small "Pending demo requests" list (read-only + a simple dismiss/mark-contacted action) reusing its current list-rendering approach; no approve/provision workflow needed since no account is created.
- **This is new backend surface, not just content** — smaller and lower-risk than it might sound (mirrors an already-hardened, already-tested public endpoint almost exactly; adds one new Durable Object key prefix, no changes to any existing key), but it does touch `cloudflare/worker.js`, which is explicitly a high-risk file in this repo's own docs. Calling this out for explicit sign-off as its own item, separate from the pure-content landing page work.
- **Deployment dependency:** the landing page itself (static, GitHub Pages) can go live independently, but the "Request a live demo" button will submit to a route that doesn't exist until the Worker is redeployed — per this repo's standing rules, `wrangler deploy` is the owner's action, never run automatically. Until that deploy happens, the demo form would error on submit. Sequencing note added to Staging/sequence and Test plan below.

## Rollback

Purely additive: new files (`landing.js`, `demo-util.js`, `demo.html`, `demo.js`), new test files, new markup, one new line in `auth.js`, a version bump. The one exception to "nothing server-side": the demo-request feature adds a new Durable Object key prefix (`demo:*`) and a new public Worker route — still purely additive (no existing key, route, or schema is modified or removed), so rollback there is also a plain revert of the Worker code; any already-stored `demo:*` entries would simply become unreachable dead keys, not corrupted or exposed data.

One nuance (confirmed by review, turndesk is live/main-only with no staging buffer): if a rollback is ever needed **after** real clients have already loaded and cached `v0.44`, a plain `git revert` that re-lowers the version string won't cleanly self-heal those clients — the service worker's update-detection compares version numbers, so undoing a bump needs a **forward** version bump (e.g. to `v0.45` carrying the old content) to actually purge the stale cache, not a literal revert of the version files. Before this change is live, rollback is trivially a plain revert (nothing has been served yet to cache).
